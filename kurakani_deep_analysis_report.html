🔴 Security
CRITICAL
Attachments stored publicly — all files world-readable
supabase/migrations/…_signal_init.sql + security_hardening.sql
▾

The initial migration creates the attachments Storage bucket with public = true and a "Allow public read access on attachments" policy. Even though the security hardening migration later flips the bucket to private, the migrations run sequentially — any deployment gap exposes all uploaded media publicly. More importantly, the fix relies on running a later migration; if any deployment step is missed, real user files are publicly accessible without authentication.

FixCreate the bucket private from day one: values ('attachments', 'attachments', false). Never use a public bucket for user content. The security hardening migration is now also redundant and can be simplified.
CRITICAL
WebSocket connects with no auth token — getAccessToken() always returns null
apps/web/src/lib/socket.ts:19
▾

The SocketManager.connect() calls getAccessToken() from api.ts — but that function was deprecated and returns null unconditionally after the Supabase migration. So if (!token) return; on line 19 prevents any WebSocket connection from ever being established. Any code path that falls back to this socket (e.g. real-time call signalling) silently does nothing.

FixReplace with const { data: { session } } = await supabase.auth.getSession(); const token = session?.access_token; before calling connect. Or migrate all real-time to Supabase Realtime channels (which already handle auth) and delete the legacy WebSocket layer entirely.
CRITICAL
Any authenticated user can mark any OTPK as used (race attack breaks E2EE)
supabase/migrations/…_signal_init.sql — one_time_pre_keys UPDATE policy
▾

The initial migration creates: CREATE POLICY "Users can update one time pre-keys (mark as used)" ON one_time_pre_keys FOR UPDATE USING (true); — any logged-in user can mark every other user's OTPKs as used. This exhausts OTPKs, forcing session establishment without forward secrecy. The security hardening migration fixes this, but only if it runs.

FixThe correct policy is in the hardening migration: USING (auth.uid() = user_id). Ensure this migration always runs before any production traffic. Also add a server-side function to atomically fetch-and-mark a key so the client never directly writes used = true.
CRITICAL
Group admin transfer has no server-side ownership check
apps/web/src/app/(app)/messages/[id]/page.tsx:544
▾

handleTransferAdmin directly updates conversation_members.role = 'admin' for any user_id via Supabase client. The RLS policy for conversation_members FOR UPDATE only checks auth.uid() = user_id (own row) — meaning any member could update their own role to admin by crafting a direct Supabase call, bypassing the UI check. The UI gate (member.role === 'admin') is purely client-side and trivially bypassed.

FixAdd an RLS policy that restricts role column updates to existing admins only: WITH CHECK (EXISTS(SELECT 1 FROM conversation_members WHERE conversation_id = ... AND user_id = auth.uid() AND role = 'admin')). Or use a Supabase Edge Function / Postgres function with SECURITY DEFINER to enforce this server-side.
HIGH
Plaintext phone number sent to Supabase auth metadata before hashing
apps/web/src/app/(auth)/register/page.tsx:47
▾

Registration passes phone_number: phoneNumber raw in supabase.auth.signUp options.data. Supabase stores this in auth.users.raw_user_meta_data. The hashing only happens inside the handle_new_user() trigger when copying to public.users.phone_hash — but the plaintext version remains in auth.users forever and is accessible via Supabase Auth admin APIs.

FixHash the phone number client-side before passing it to Supabase, or omit it entirely from auth metadata and pass only a pre-hashed value. Also set the default pepper via ALTER DATABASE postgres SET app.phone_pepper = '...' in a secrets-managed migration instead of hardcoding 'kurakani_default_pepper'.
HIGH
Mobile app uses a completely separate, legacy JWT auth system
apps/mobile/src/lib/api.ts, apps/mobile/src/screens/*
▾

The mobile app stores signal_token and accessToken in unencrypted AsyncStorage and hits a separate Express backend at localhost:4000. The web app has migrated fully to Supabase, but the mobile app hasn't. This means: (1) two separate auth systems with different sessions and session invalidation, (2) JWT tokens stored in AsyncStorage (not the OS secure enclave), (3) the backend at port 4000 may not exist in production, silently failing all mobile API calls.

FixMigrate mobile to Supabase Auth (@supabase/supabase-js for React Native). Use react-native-encrypted-storage (already used in SignalStore) for session tokens instead of AsyncStorage. Remove the Express backend dependency from mobile entirely.
HIGH
conversation_members SELECT policy is infinitely recursive
supabase/migrations/…_signal_init.sql
▾

The policy reads: USING (EXISTS (SELECT 1 FROM conversation_members cm WHERE cm.conversation_id = conversation_members.conversation_id AND cm.user_id = auth.uid())). The subquery references the same table that the policy is on — Supabase/Postgres will detect this and return an error or silently break. This is the classic RLS self-referential recursion bug.

FixUse a security-definer helper function or restructure: USING (EXISTS (SELECT 1 FROM conversation_members cm JOIN conversations c ON c.id = cm.conversation_id WHERE c.id = conversation_members.conversation_id AND cm.user_id = auth.uid())) — or better, set SET search_path = public and use a SECURITY DEFINER view.
HIGH
File upload: no server-side MIME type validation, content-type trusted from client
apps/web/src/app/(app)/messages/[id]/page.tsx:571
▾

uploadToS3 passes file.type (client-supplied MIME string) directly as the Supabase Storage contentType. An attacker can set file.type = "text/html" on a JavaScript payload and upload it. The storage bucket allows image/*, video/*, audio/* via the file input UI — but that's a browser-only guard. Since attachments render with <img src={m.mediaUrl}> and <audio src={m.mediaUrl}>, XSS via stored HTML/SVG is a real vector if the bucket allows direct serving.

FixValidate MIME type server-side via a Supabase Storage Transform or Edge Function that inspects the file magic bytes. Restrict the bucket's allowed MIME types. Ensure the CDN serves files with Content-Disposition: attachment for non-image types. Never render mediaUrl as <img> without validating the URL and MIME type.
🟡 Logic bugs
LOGIC
Every login regenerates Signal identity — breaks E2EE history for all existing sessions
apps/web/src/app/(auth)/login/page.tsx:58–82
▾

On every login the app calls generateSignalRegistrationPayload(store) and then inserts fresh identity_keys, signed_pre_keys, and 100 new one_time_pre_keys. This is wrong: identity keys must be stable across sessions. Regenerating them means: (1) all existing sessions with contacts are broken — contacts have the old identity key and will fail to decrypt, (2) 100 new OTPKs flood the DB on every login, (3) the INSERT on identity_keys fails silently because of the UNIQUE(user_id, device_id) constraint and the code logs a warning but continues.

FixOn login, check if the local WebSignalStore.isInitialized(). If yes, skip key generation. Only generate a new identity if the store is empty (first device use). Restore keys from a backup if the store is cleared. The registration flow (new account) correctly generates keys; login should reuse them.
LOGIC
Sender's own past messages always show as "Sent Message" — history is lost
apps/web/src/app/(app)/messages/[id]/page.tsx:317
▾

In loadMessages(), when iterating history: if (m.sender_id === currentUserId) { payload = { content: 'Sent Message', media_url: null }; }. Since Signal E2EE is one-directional (you encrypt to the recipient's keys, not your own), the sender can't decrypt their own messages from the ciphertext in the database. But showing "Sent Message" placeholder breaks UX entirely — past conversations look like gibberish. The same bug exists in loadMoreMessages().

FixStore a plaintext copy (or a copy encrypted to the sender's own key) of sent messages in the local Dexie DB at send time. On handleSendMessage, immediately persist the plaintext to db.local_messages before sending (already done for the optimistic add). The loadMessages function should query Dexie first, and only fall back to decryption for received messages.
LOGIC
Group chats are architecturally broken — encryption only works for 1-on-1
apps/web/src/app/(app)/messages/[id]/page.tsx:183, 505
▾

const other = conv.members.find((m: any) => m.id !== currentUserId) picks the first non-self member. In a group of 3+, this is only one person. Encryption then calls encryptMessage(store, otherUser.id, 1, payloadString) — encrypting to only one recipient. All other group members receive ciphertext they cannot decrypt. The Group Settings modal also appears in the UI but returns "Group invites not supported in MVP" — the feature is partially built but non-functional.

FixEither (a) disable group chats in the UI and remove the dead UI code, or (b) implement proper group E2EE using Signal's Sender Key mechanism: generate a group sending chain, distribute the sender key to all members, and encrypt each message once to the group key. This is a significant feature, not a quick fix.
LOGIC
Typing indicators broadcast on a new unsubscribed channel instance
apps/web/src/app/(app)/messages/[id]/page.tsx:443
▾

The useEffect for real-time creates a channel: const channel = supabase.channel('room:' + conversationId). But typing indicator sends use a separate call: supabase.channel('room:' + conversationId).send(...) — this creates a new, unsubscribed channel instance each time. Supabase channels must be subscribed before you can broadcast. The typing events are silently dropped.

FixStore the channel reference in a useRef: const channelRef = useRef(null). Assign on creation: channelRef.current = supabase.channel(...).subscribe(). In handleInputChange use channelRef.current?.send(...).
LOGIC
sanitizeMessage() is defined but never called — plaintext renders unsanitized
apps/web/src/lib/sanitize.ts vs apps/web/src/app/(app)/messages/[id]/page.tsx
▾

A DOMPurify sanitizer is correctly implemented in sanitize.ts, but the chat thread renders messages as: <p className="font-sans">{m.plaintext}</p> — raw JSX text interpolation, bypassing sanitization entirely. Because React escapes plain text children by default this prevents innerHTML XSS, but it also means decrypted HTML tags appear literally as text rather than being stripped. Any intentional rich-text rendering would need this.

FixIf rich text is desired: use dangerouslySetInnerHTML={{ __html: sanitizeMessage(m.plaintext) }}. If it's not desired, keep plain text and remove sanitize.ts to avoid confusion. Also: the sanitizeMessage function returns '' on the server (SSR guard) — this needs a fallback for server rendering.
LOGIC
OTPK race condition — two simultaneous sessions can grab the same key
apps/web/src/app/(app)/messages/[id]/page.tsx:488–509
▾

Session establishment fetches an OTPK with .eq('used', false).limit(1) and then marks it used in a separate request. Two users opening a chat with the same person at the same time both get the same OTPK, and both mark it used after the fact. This violates the one-time property of OTPKs and weakens forward secrecy.

FixUse a Postgres function with FOR UPDATE SKIP LOCKED to atomically fetch-and-mark a key in one transaction: SELECT id, key_id, public_key FROM one_time_pre_keys WHERE user_id=$1 AND used=false LIMIT 1 FOR UPDATE SKIP LOCKED, then immediately update in the same transaction. Expose this as a Supabase RPC.
🔵 Code quality & performance
PERF
searchUsers() interpolates query into .or() — potential ReDoS / injection
apps/web/src/lib/api.ts:88
▾

.or('username.ilike.%${query}%,display_name.ilike.%${query}%') — the query string is interpolated directly. Supabase's .or() parses this string server-side. A user could inject malformed filter syntax (id.eq.*, escaping the % context). Additionally, leading/trailing wildcard %query% means a two-character search scans the entire table without an index. The username column has an index, but ilike with leading wildcard disables it.

FixSanitize the query: strip special characters (%, _, backslash) before interpolation. Consider using .textSearch() (full-text search) or a LIKE 'query%' prefix match instead of %query% to use indexes. Add a minimum query length of 2 characters before firing the search.
QUALITY
arrayBufferToBase64 is duplicated in 3 separate files
packages/crypto/src/keys.ts, packages/crypto/src/session.ts, apps/web/src/lib/crypto/registration.ts
▾

The same ~6-line arrayBufferToBase64 function is copy-pasted in three places. This is the shared crypto package — the function should be exported once from packages/crypto/src/index.ts and imported everywhere else.

FixExport from packages/crypto/src/index.ts: export function arrayBufferToBase64(buf: ArrayBuffer): string. Use import { arrayBufferToBase64 } from '@signal/crypto' in all consumers. Also note: the browser-native Uint8Array.prototype + btoa approach works but is slow for large buffers — consider Buffer.from(buf).toString('base64') in Node or the TextDecoder approach in browsers.
QUALITY
Sealed sender and attachment encryption are fully implemented but never used
packages/crypto/src/sealed-sender.ts, apps/web/src/lib/crypto/attachments.ts
▾

The sealed-sender.ts module (which hides sender identity from the server) and the attachments.ts (which encrypts file data before upload) are well-implemented but have zero call sites. Attachments are uploaded in plaintext to Supabase Storage — their URLs are sent as media_url inside the E2EE message, but the file bytes themselves are unencrypted.

FixFor attachments: call encryptAttachment(fileBytes) before uploadToS3, upload the encrypted bytes, then send the key+IV inside the Signal message payload. For sealed sender: integrate into the message send flow if sender anonymity is a design goal. If not, document that it's planned and add a TODO so it's not forgotten.
QUALITY
Schema fragmentation — two separate parallel schema evolutions with DROP TABLE conflicts
supabase/migrations/ (4 migration files)
▾

The migrations folder has two separate lineages: (1) a social platform schema (profiles, follows, posts, likes, comments) and (2) a messaging schema (users, conversations, messages) — and both start with DROP TABLE IF EXISTS messages CASCADE; DROP TABLE IF EXISTS conversations CASCADE;. Running all migrations in order will drop and recreate tables multiple times. The profiles table and users table are separate but both have triggers mapping from auth.users — two handle_new_user() functions that conflict.

FixConsolidate into one authoritative schema. Decide whether profiles or users is the source of truth for user data (not both). Delete the conflicting social-platform migration or merge it. Ensure migration files are numbered and ordered correctly for idempotent replay.
QUALITY
password_hash column stored in users table as 'supabase_auth' dummy value
supabase/migrations/…_auth_trigger.sql
▾

The users table has a password_hash VARCHAR(255) NOT NULL column. Since Supabase Auth handles passwords, the trigger inserts 'supabase_auth' as a dummy value. This is confusing (someone might think it's the real hash), wastes space, and makes the schema misleading. A later migration drops the NOT NULL constraint but the column remains.

FixDrop the password_hash column entirely: ALTER TABLE users DROP COLUMN password_hash;. Supabase Auth is the source of truth for credentials — there's no need to duplicate or shadow it in public.users.
QUALITY
No password complexity or minimum length validation on registration
apps/web/src/app/(auth)/register/page.tsx
▾

The password field has required only — no minlength, no pattern, no strength check. Supabase's default minimum is 6 characters. A user can register with password "123456" with no warning. For a security-focused E2EE app this is a significant UX/security gap.

FixAdd minLength={12} to the password input, a real-time strength meter, and enforce minimum requirements in Supabase Auth settings (Dashboard → Auth → Password strength). Consider passphrase guidance since the app already uses "passphrase" branding.
QUALITY
Legacy auth shim with deprecated no-ops keeps dead code in the critical path
apps/web/src/lib/api.ts:1–14, apps/web/src/lib/socket.ts:1
▾

setTokens, getAccessToken, getRefreshToken, clearTokens are all exported as @deprecated no-ops. socket.ts still imports and calls getAccessToken() which returns null, breaking the WebSocket connection silently. Dead deprecated code in production is a maintenance hazard and caused a real bug here.

FixDelete all deprecated shims. Fix socket.ts to use Supabase session token directly. Run a codebase-wide search for any other consumers of the deprecated functions and update or remove them.
